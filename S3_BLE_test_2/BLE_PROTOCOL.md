# BLE Protocol for Client

Firmware now uses one BLE service with:

- Service UUID: `4fafc201-1fb5-459e-8fcc-c5c9c331914b`
- ALERT characteristic   (notify, read): `beb5483e-36e1-4688-b7f5-ea07361b26a8`
  carries **`ALERT`** and **`SAFE`** packets
- VITALS characteristic  (notify, read): `7b809f11-63f0-4dca-8e4d-2b4e8384e7c1`
  carries **`BATCH`** (HR/SpO2) and **`BMI`** (BMI160 live peak) packets
- CONTROL characteristic (read, write):  `f9b2c417-1d15-4ad4-9b52-b94aa0f76b03`
  client writes `READY` after subscribe; device replies `ACK:READY`

The old continuous IMU telemetry stream is no longer used. Live BMI160 activity
is summarised by the `BMI` packet emitted every 5 s (peak acc + peak gyro).

## Packet data-source legend

| Packet | Source                                  | Real / Simulated |
| ------ | --------------------------------------- | ---------------- |
| ALERT  | BMI160 + TFLite V84 fall pipeline       | **Real**         |
| SAFE   | Hardware button (GPIO 10) during alarm  | **Real**         |
| BMI    | BMI160 sensor peak in last window       | **Real**         |
| BATCH  | HR / SpO2 generator (`readHrSample()` / | Simulated *      |
|        | `readSpo2Sample()` in `src/main.cpp`)   |                  |

\* `255` is the documented sentinel for "sensor not ready" — the Android parser
maps it to `-1` so the UI never displays a fake reading. When a real PPG driver
is integrated, only the two `readHrSample()` / `readSpo2Sample()` functions need
to change; the BLE payload contract stays identical.

## 1. Connection model

- The ESP32 keeps collecting data locally at the edge.
- `fall` is sent immediately when detected.
- Vitals are sampled every `5s`.
- Every `25s`, the firmware creates one batch containing:
  - `HR[5]`
  - `SpO2[5]`
  - `timestamp[5]`
- If no BLE client is connected, those packets are stored in internal RAM queues.
- When a client connects, the ESP32 does not flush immediately.
- The client must:
  - subscribe to `status`
  - subscribe to `vitals`
  - write `READY` to the control characteristic
- Only after `READY`, the ESP32 flushes queued packets in FIFO order:
  - all queued `ALERT` packets first
  - then queued `BATCH` packets

## 2. Recommended client behavior

- Connect to the service.
- Request MTU as large as possible.
  - Recommended: `>= 128`
  - The current payloads are ASCII strings and are much safer with a larger MTU.
- Subscribe to both notify characteristics before processing data.
- After subscribing, write `READY` to the control characteristic.
- Treat notifications as append-only events.
- Reconnect automatically on disconnect.
- Deduplicate packets by `sequence` number.

## 2.1. READY handshake

Client write:

```text
READY
```

Purpose:

- tells ESP32 that notify subscriptions are already active
- prevents ESP32 from flushing backlog too early
- ensures queued packets are delivered after the client is actually listening

## 3. Fall alert payload

Format:

```text
ALERT,<sequence>,<timestamp_sec>,fall,<status_code>,<fall_prob>,<non_fall_prob>
```

Example:

```text
ALERT,12,845,fall,1,0.873,0.127
```

Fields:

- `sequence`: monotonically increasing alert id generated by firmware
- `timestamp_sec`: device uptime in seconds when the fall was emitted
- `fall`: literal event label
- `status_code`: currently `1` for fall
- `fall_prob`: model probability for fall
- `non_fall_prob`: model probability for non-fall

Important:

- `timestamp_sec` is currently based on `millis()/1000`, so it is device uptime, not wall-clock UTC.
- If you need real-world timestamps on the client, map packet arrival time or sync the device clock in a future revision.

## 4. Vitals batch payload

Format:

```text
BATCH,<sequence>,<hr0>|<hr1>|<hr2>|<hr3>|<hr4>,<spo20>|<spo21>|<spo22>|<spo23>|<spo24>,<ts0>|<ts1>|<ts2>|<ts3>|<ts4>
```

Example:

```text
BATCH,31,72|74|75|73|76,98|98|97|98|97,810|815|820|825|830
```

Fields:

- `sequence`: monotonically increasing batch id
- `hr*`: 5 heart-rate samples
- `spo2*`: 5 blood oxygen samples
- `ts*`: 5 timestamps in UTC epoch seconds

Semantics:

- Consecutive timestamps should normally differ by `5`.
- Each batch contains exactly `5` samples.
- A batch is emitted every `25s`.

## 5. SAFE payload (device button)

Format:

```text
SAFE,<sequence>,<timestamp_sec>
```

Emitted only when the user presses the device button **during an active fall
alarm**. Resets the on-device alarm and tells the app to cancel its 15 s
emergency countdown (equivalent to the user pressing "I'm Safe" in the app).

## 6. BMI live snapshot payload

Format:

```text
BMI,<sequence>,<timestamp_sec>,<peak_acc_g>,<peak_gyro_dps>,<active>
```

Example:

```text
BMI,42,123,1.034,5.7,0
BMI,43,128,3.210,287.4,1
```

Fields:

- `sequence`: monotonically increasing snapshot id
- `timestamp_sec`: device uptime seconds
- `peak_acc_g`: max |acc| magnitude over the latest 2 s inference window, in g
- `peak_gyro_dps`: max |gyro| magnitude over the latest 2 s window, in dps
- `active`: `1` if the window crossed the activity threshold
  (acc > 2 g OR gyro > 50 dps), `0` otherwise

Cadence: every **5 s**. This is REAL BMI160 data — useful for showing the
caregiver that the device is alive and that motion is being observed.

## 7. Missing data semantics

Current firmware does not contain a real HR/SpO2 sensor driver, so
`BATCH` HR and SpO2 values come from a simulator. The simulator is
deliberately well-behaved (60–100 bpm, 94–100 %).

If the future driver returns "sensor not ready" it MUST emit `255`. The
Android parser maps `255 → -1` and the UI hides that sample. No other
client/firmware change is required when the real driver lands.

## 6. Suggested parser logic

Pseudo-flow:

```text
if payload starts with "ALERT,":
    split by ","
    parse sequence, timestamp_sec, label, status_code, fall_prob, non_fall_prob
    ignore if sequence already processed
    trigger emergency workflow

if payload starts with "BATCH,":
    split by ","
    parse sequence
    split hr section by "|"
    split spo2 section by "|"
    split timestamp section by "|"
    ignore if sequence already processed
    for i in 0..4:
        if hr[i] != 255 and spo2[i] != 255:
            save sample
        else:
            mark sample invalid
```

## 7. Storage recommendation on client

For each batch:

- store the raw packet
- store the parsed 4 points as independent records
- include:
  - `device_id`
  - `batch_sequence`
  - `sample_index`
  - `heart_rate`
  - `spo2`
  - `timestamp_sec`
  - `received_at`

This lets you:

- render charts easily
- deduplicate by sequence
- rebuild a time series even after reconnect

## 8. Reliability note

Current firmware flushes queued packets only after the client sends `READY`, but it does not yet implement per-packet application-level ACK.

That means:

- it is good enough for most BLE client apps
- it is not a strict guaranteed-delivery protocol

If you later want stronger delivery guarantees, add:

- one write characteristic from client to device
- ACK payload: `ACK,<type>,<sequence>`

Then the firmware can delete queued packets only after explicit ACK.
